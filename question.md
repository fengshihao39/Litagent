# 星火杯答辩 - 可能出现的提问与参考回答

---

## 一、系统整体设计

**Q：你们这个系统的核心功能是什么？**

A：Litagent 是一个科研文献智能助手，核心功能有五块：
1. 多数据源论文搜索（arXiv、Semantic Scholar、Crossref）
2. 论文详情展示（引用数、热度评分、作者、关键词）
3. PDF 全文下载与解析，配合 AI 生成深度结构化总结
4. 参考文献和施引文献查询
5. 搜索历史与收藏管理

---

**Q：为什么要做这个？和直接用 Google Scholar 有什么区别？**

A：Google Scholar 主要做的是"找到论文"，但找到之后怎么快速读懂？需要自己打开 PDF 一页页读。
Litagent 在"找到"的基础上增加了"读懂"的能力：自动下载 PDF、解析全文、用 AI 生成结构化总结（研究背景、方法、结论等），科研人员不需要切换多个平台，也不需要自己逐字阅读。

---

**Q：你们整体用了什么技术栈？**

A：
- 后端：Python + FastAPI
- 前端：Streamlit（快速原型，适合 AI 应用展示）
- AI 大模型：DeepSeek API（deepseek-chat）
- PDF 解析：PyMuPDF（fitz）
- 数据库：SQLite（本地缓存历史记录）
- 文献数据源：arXiv、Semantic Scholar、Crossref、OpenAlex
- 包管理：uv

---

## 二、AI 总结 / LLM 相关

**Q：你们是怎么让 AI 读论文的？**

A：分两步：
1. 用 PyMuPDF 把 PDF 里的文本抽取出来，并按章节（Abstract、Introduction、Method、Experiment、Conclusion）切分整理
2. 把整理好的全文片段（约 12000 字以内）加上论文元数据，发给 DeepSeek 生成结构化总结

AI 并不是直接"看"PDF，而是在读文本提取结果。

---

**Q：你们用的 prompt 是怎么设计的？**

A：Prompt 分两套：
- 全文模式（有 PDF 文本）：要求 AI 输出 8 个模块，包括研究背景、研究问题、核心方法、实验设置、主要结论、创新点、应用场景、局限性，要求严格基于原文，不得编造
- 摘要模式（没有 PDF）：简化为 6 个模块，允许 AI 基于摘要推断，但必须注明"AI 推断，非全文依据"

这样设计是为了保证输出质量的同时，对两种情境都能覆盖。

---

**Q：LLM 会不会编造论文里没有的内容（幻觉问题）？**

A：会有风险，我们做了几个缓解措施：
1. Prompt 明确要求"只能基于原文"，找不到的信息要写"论文未明确说明"
2. 全文模式下喂给模型的是真实论文文本，不是模型自己生成的
3. AI 推荐的参考文献会单独标注"AI 推荐·待核验"，和真实数据库结果分开展示
4. 摘要模式的推断内容也会标注来源

幻觉问题在任何 LLM 应用中都存在，我们的处理思路是"透明化"而非"假装没有"。

---

**Q：为什么选 DeepSeek 而不是 GPT-4 或 Claude？**

A：主要考虑三点：
1. 成本：DeepSeek 价格较低，适合原型开发和比赛阶段
2. 中文能力：DeepSeek 对中文学术场景支持好
3. API 兼容性：兼容 OpenAI SDK，集成方便、迁移成本低

后续如有需要，模型部分可以很方便地替换成其他 LLM，架构上没有强绑定。

---

## 三、PDF 解析相关

**Q：你们是怎么读 PDF 的？**

A：用 PyMuPDF 逐页提取文本层，不是 OCR，不是截图分析。具体是：
1. `fitz.open()` 打开 PDF
2. 遍历每一页调用 `page.get_text()`
3. 拼接成全文字符串
4. 用正则按章节标题切分成有结构的片段
5. 每个 section 限制约 2000 字，总量控制在 12000 字内，再发给 LLM

---

**Q：那扫描版 PDF 能处理吗？**

A：目前不支持扫描版。PyMuPDF 读的是 PDF 的文本层，扫描版本质上是图片，没有文本层，提取出来的字符数会非常少（系统判断少于 200 字则认为失败）。
这种情况下系统会自动降级到摘要模式，不会报错崩溃。
如果后续要支持扫描版，可以接入 OCR（如 PaddleOCR）。

---

**Q：双栏论文处理效果怎么样？**

A：双栏论文是已知局限。`page.get_text()` 没有做版面分析，可能出现左右栏文本交叉读取的情况。
在实际测试中，对 arXiv 上常见的标准双栏论文，做快速总结通常还能使用，但如果要严格按段落顺序精读，现阶段这个版本不够稳。
这是后续可以用 LayoutParser 或版面分析模型来升级的方向。

---

**Q：为什么选 PyMuPDF 而不是其他 PDF 库？**

A：主要是三点：
1. 速度快，纯 C 底层，处理一篇论文 PDF 通常在 1 秒以内
2. API 简洁，`page.get_text()` 一行搞定文本提取
3. 对学术论文（LaTeX 生成的 PDF）的文本层保留较好，提取质量在同类库中较有竞争力

和 pdfminer、pdfplumber 相比，速度优势明显；和 pypdf 相比，文本质量更稳。

---

## 四、数据来源相关

**Q：你们用了哪些数据源？**

A：
- arXiv：预印本论文，开放获取，PDF 直链稳定
- Semantic Scholar：学术搜索，提供引用数、参考文献、施引文献关系
- Crossref：出版商数据，提供 DOI、ISSN、期刊名、发表类型等元数据
- OpenAlex：开放学术知识图谱（备选）

---

**Q：Semantic Scholar 会限流，你们怎么处理？**

A：会的，Semantic Scholar 公开 API 限流较严。我们的处理是：
1. 请求之间加了适当延迟（`time.sleep()`）
2. 查不到或 429 时静默降级，返回空结果而不是报错
3. 部分结果有本地缓存，减少重复请求

申请了 API Key 后限流会宽松很多，目前阶段属于已知限制。

---

**Q：参考文献不够的时候你们是怎么补的？**

A：当 Semantic Scholar 返回真实参考文献少于 3 条时，会让 DeepSeek 根据论文标题和摘要推断 2 条可能的参考文献。
这 2 条 AI 推荐的条目会单独展示，并标注"AI 推荐·待核验"，和真实数据库的结果明确分开，不会混入正式参考文献列表。
同时，AI 推荐的条目还会再经过 Semantic Scholar 搜索复核，如果能找到真实数据就用真实数据替换。

---

## 五、系统架构相关

**Q：为什么前端用 Streamlit 而不是 React？**

A：比赛阶段追求快速验证功能，Streamlit 可以用纯 Python 构建交互界面，不需要写 HTML/CSS/JS，开发效率非常高。
对于 AI 应用的 demo 展示，Streamlit 已经成为事实上的标准工具（Hugging Face 上大量 demo 都用它）。
如果要上生产或追求更好的 UX，可以换成 React + FastAPI 前后端分离架构，后端接口设计不需要大改。

---

**Q：数据库为什么用 SQLite 而不是 MySQL/PostgreSQL？**

A：
1. 比赛场景下不需要多用户并发，SQLite 够用
2. 零配置、零部署，文件即数据库，本地运行方便
3. Python 标准库内置支持，不需要额外依赖

如果后续需要多用户或云端部署，迁移到 PostgreSQL 只需要改存储层，业务逻辑不需要动。

---

**Q：你们的后端接口是怎么设计的？**

A：用 FastAPI，RESTful 风格。主要接口分几类：
- `/search`：多数据源论文搜索
- `/paper/*`：论文详情、AI 总结、参考文献、施引文献、期刊信息、翻译
- `/history/*`：搜索历史、收藏管理

新功能统一加新接口，不修改已有接口，保证向后兼容。

---

## 六、比较难回答的深度问题

**Q：你们系统和 Semantic Scholar 本身有什么区别？**

A：Semantic Scholar 是数据源，我们是在它之上做的应用层。
Semantic Scholar 提供数据，但不会帮你：
- 下载 PDF 全文
- 用 AI 解读论文
- 把研究背景、方法、结论提炼成结构化摘要
- 在一个界面里整合多源数据、历史记录、收藏

我们做的是"从找到到读懂"这个增值链路。

---

**Q：如果论文是中文的，你们支持吗？**

A：目前 PDF 解析的章节切分逻辑是针对英文学术写作规范设计的（Abstract、Introduction 等）。中文论文如果章节结构类似，勉强能用；但专门针对中文论文的版面识别和章节切分目前还没做。
AI 总结本身（DeepSeek）支持中文，所以如果拿到了正确的中文文本，总结是可以生成中文的。

---

**Q：你们有没有测试过准确率？**

A：目前是功能验证阶段，没有做系统性的量化评估。
在测试过程中，对 arXiv 上的英文论文，AI 总结的覆盖度和相关性主观评价较好；参考文献的 Semantic Scholar 召回率受论文收录情况影响，约有 60~70% 的论文能查到参考文献。
后续可以设计 Recall@K 等指标做更系统的评估。

---

**Q：这个系统实际使用有什么局限？**

A：主要有三点：
1. 依赖外部 API，Semantic Scholar 限流、Crossref 慢时体验下降
2. PDF 解析对扫描件、复杂版式支持有限
3. LLM 存在幻觉风险，输出内容仅供参考，不能替代原文阅读

这些局限我们在界面上都做了标注和降级处理，而不是隐藏。

---

**Q：下一步打算怎么做？**

A：有三个方向：
1. 增加 OCR 支持，覆盖扫描版 PDF
2. 增加版面分析（如 LayoutParser），提升双栏论文解析质量
3. 增加论文知识图谱功能，把引用关系可视化展示
